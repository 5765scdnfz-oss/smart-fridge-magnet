## 一、背景：100天旗帜计划1.0

去年给自己定了一个100天旗帜计划，希望通过100天的时间完成健康生活状态的回归，同时了解AI工具的边界。

**1.0方案：**
- Google全家桶：Gemini模型 + Google Drive + Pixel手机
- 三个月减了20多斤：86公斤 → 74公斤
- 大部分减掉的是脂肪，效果成功

**为什么要做2.0：** 以前的方案不够好，要做一回自己的产品经理。

## 二、需求解构

在减肥过程中，脑中想象到一个终极画面：

> **能不能我只负责生活，然后AI真正去替我解决在这个反人类的减肥目标面前所有的使用成本和用户卡点？**

### 2.1 三大核心需求

| 需求 | 描述 |
|------|------|
| **输入极其简单** | 不需要知道每盘菜多少克，拍个照片/打个字/语音描述就能记录热量 |
| **营养知识支持** | AI提供专业的营养学知识（为什么减肥要做力量而不是有氧等） |
| **自动化无感知闭环** | 自动获取消耗数据 + 记录摄入数据 → 形成清晰闭环 + 趋势总结 |

### 2.2 市面上产品的痛点

1. **所有软件都需要填表** — 薄荷、FatSecret等都是手动输入
2. **欧美开发者设计** — 基于白人饮食结构，不适合中国复杂餐饮
3. **食材库建立困难** — 大包套小包的预制食材难以标准化
4. **数据闭环断裂** — 手表数据无法自动同步，每晚12点要手动截图

## 三、技术选型

### 3.1 核心原则：Think Less

整个前端是一层数据投影，尽可能干掉需要界面解决的问题：

| 原则 | 做法 |
|------|------|
| **LUI优先** | Agent优先用自然语言界面，通过聊天建Profile，不需要手动表单 |
| **UI只做展示** | 环形图、趋势线图等结构化展示，不做手动设定 |
| **干掉卡片** | 用Markdown能解决的就不要做卡片，降低开发成本 |

### 3.2 语言选择：React Native

- 之前做日历项目用Swift，走出舒适区选择React Native
- 开发效率高很多，热加载及时看到前端调整结果
- Expo + EAS部署，个人使用完全够用

### 3.3 Agent Runtime：本地优先

**为什么放在本地（Mac Studio）：**

| 优势 | 说明 |
|------|------|
| **开发运维一体** | 开发、运行、维护都在同一台机器 |
| **远程Debug** | 通过Discord远程告诉agent查看生产日志，出差路上就能排查问题 |
| **云同步** | agent读取本地macOS日历/日志，自动同步训练计划 |
| **读写效率** | 本地数据库读取一毫秒不到，API级别稳定 |

> 本质就是：一个本地的Local Agent Runtime，充分利用桌面操作系统的权限。

### 3.4 API选择

| 用途 | 选择 | 原因 |
|------|------|------|
| **推理** | Anthropic Claude（Haiku + Sonnet） | 工具调用稳定，推理效果好 |
| **Embedding** | 阿里通义Embedding V4 | 1000+食材库向量化检索，便宜够用 |
| **健康数据** | ROOK API | 聚合Garmin/Strava等数据源，个人免费 |

### 3.5 SDK选择：Claude Agent SDK

- 比Vercel SDK完成度更高
- 基于Claude Code框架打造
- **定制化：** 把Tool Runner从串型改成并行（一顿饭可以同时记录多个食材）

### 3.6 上下文工程

每次Agent回答问题前，先加载：
- 整个数据库
- 今天摄入情况
- 过去一周的数据预热

**效果：** 不是简单说"你吃多了"，而是"今天晚餐虽然多了点，但这周还有2000多热量缺口，偶尔放松有利于长期坚持"。

### 3.7 可观测性

- 每轮对话的Transcript记录
- 每次工具调用的日志和TryCatch
- Token消耗和成本追踪
- 数据库读写记录

## 四、食材结构化设计

### 4.1 两大类食材

| 类型 | 说明 | 示例 |
|------|------|------|
| **天然食材** | 基本营养成分：热量、脂肪、碳水、蛋白质、纤维 | 鸡胸肉、西兰花 |
| **预制食材** | 大包套小包，按小包分量记录 | 饺子（猪肉白菜 vs 芹菜牛肉热量不同） |

### 4.2 热量估算方法

**发现：** 世界上不存在绝对准确的食材热量评估方式，100%都是估算。

**解决方案：**

1. **AI估算** — 常见食材估得蛮准（鸡胸肉一般100-120g）
2. **国家营养学会数据** — 花钱买了中国1700种常见食材营养表
3. **烹饪系数** — 蒸×1.1，炸×1.9~2.5以上
4. **参照系校准** — 拍照时旁边放参照物，结合中国烹饪系数

### 4.3 中国饮食特点

- 最不缺碳水（小麦和水稻摄入多）
- 不缺脂肪（煎炒烹炸太多）
- 烹饪方式复杂多样，不同于欧美简单的沙拉/烤箱

## 五、数据闭环方案

### 5.1 核心痛点

手表数据 → Garmin Connect → Garmin服务器 → ？？？

**Garmin数据获取方式：**

| 方式 | 问题 |
|------|------|
| 成为Garmin开发者 | 需要商用授权，个人不行 |
| 第三方API（Strava等） | 太贵，几百美金/月 |
| HealthKit | 苹果同步慢，Garmin选择性开放数据 |
| Apple Watch | 热量估算偏差离谱 |
| 华为/OPPO/荣耀 | 都需要成为企业开发者 |

**最终方案：** ROOK API
- 聚合Garmin/Strava/Oura等数据源
- 个人使用免费
- 授权后完整接入

### 5.2 定时任务设计

| 时间 | 任务 |
|------|------|
| 每天0:30 | 自动记录前一天的营养摄入数据 |
| 每天6:00-7:00 | 根据起床状态触发，拉取睡眠数据，规划今天的训练和饮食 |

### 5.3 中美差异

| 对比项 | 美国 | 中国 |
|--------|------|------|
| 商品条形码 | 强制绑定营养成分表 | 只绑定厂家和品类信息 |
| 扫码获取数据 | 准确 | 不准确 |
| 食材标准化 | 简单（沙拉/烤箱） | 复杂（煎炒烹炸各菜系） |
