## 一、项目概述

### 1.1 核心痛点

每天不知道吃什么、做什么，冰箱里有什么也不清楚。

### 1.2 解决方案

通过AI对话 + 拍照识别，实现：
1. 拍照自动识别冰箱食材（种类、数量、保质期）
2. 根据家庭情况 + 库存 + 口味偏好，智能推荐菜谱
3. 确认菜谱后自动扣减库存

### 1.3 我的职责范围

```
拍照 → 识别食材 → 存入库存
        ↓
家庭画像 + 库存 + 时间 → 推荐菜谱（A/B方案）
        ↓
确认方案 → 扣减库存
```

## 三、数据库设计

### 3.1 表结构

#### 表1：家庭画像（family_profile）

```sql
CREATE TABLE family_profile (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    member_name TEXT NOT NULL,        -- 成员名称（爸爸/妈妈/孩子等）
    height REAL,                       -- 身高（cm）
    weight REAL,                       -- 体重（kg）
    age INTEGER,                       -- 年龄
    allergies TEXT,                    -- 过敏食材（JSON数组）
    dislikes_main TEXT,               -- 忌口-主材料（JSON数组，如：["猪肉","牛肉"]）
    dislikes_ingredient TEXT,         -- 忌口-配料（JSON数组，如：["香菜","葱"]）
    dislikes_taste TEXT,              -- 忌口-口味（JSON数组，如：["辣","苦"]）
    health_notes TEXT,                -- 健康备注（如：糖尿病、高血压等）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### 表2：冰箱库存（fridge_inventory）

```sql
CREATE TABLE fridge_inventory (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,                -- 食材名称
    category TEXT NOT NULL,            -- 分类（蔬菜/肉类/乳制品/调味品/主食等）
    quantity REAL NOT NULL,            -- 数量
    unit TEXT DEFAULT '个',            -- 单位（个/克/盒/袋等）
    production_date TEXT,              -- 生产日期
    expiry_date TEXT,                  -- 保质期/到期日
    confidence TEXT DEFAULT '高',       -- 识别置信度（高/中/低）
    photo_path TEXT,                   -- 原始照片路径
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### 表3：用餐计划（meal_plan）

```sql
CREATE TABLE meal_plan (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    meal_type TEXT NOT NULL,           -- 餐次（早餐/午餐/晚餐）
    meal_time TEXT NOT NULL,           -- 计划用餐时间（如：18:00）
    people_count INTEGER NOT NULL,     -- 用餐人数
    plan_a TEXT,                       -- A方案菜谱（JSON）
    plan_b TEXT,                       -- B方案菜谱（JSON）
    selected_plan TEXT,                -- 用户选择的方案（A/B）
    status TEXT DEFAULT 'pending',     -- 状态（pending/confirmed/completed）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### 表4：对话历史（conversation_history）

```sql
CREATE TABLE conversation_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL,          -- 会话ID
    role TEXT NOT NULL,                -- 角色（user/assistant）
    content TEXT NOT NULL,             -- 对话内容
    action TEXT,                       -- 动作类型（识别/建档/推荐/扣减等）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 3.2 忌口分层逻辑

| 层级 | 字段 | 处理方式 | 示例 |
|------|------|----------|------|
| **主材料** | `dislikes_main` | 完全避开，不推荐含此食材的菜 | 不吃猪肉 → 不推荐红烧肉 |
| **配料** | `dislikes_ingredient` | 菜可以推荐，但标注去掉该配料 | 不吃香菜 → 推荐时标注"不放香菜" |
| **口味** | `dislikes_taste` | 调整做法或推荐替代菜 | 不吃辣 → 麻婆豆腐改为家常豆腐 |

## 五、核心流程

### 5.1 流程一：首次建档

```
用户打开App
    ↓
Agent: "你好！我是你的智能冰箱助手。先了解一下你家的情况吧：
        1. 家里几口人？
        2. 每个人有什么忌口？
        3. 平时几点吃早/午/晚餐？"
    ↓
用户: "3口人，爸爸不吃辣，孩子不吃苦瓜和葱，
       早餐7点，午餐12点，晚餐6点"
    ↓
Agent: (调用MiMo-pro提取结构化数据)
    ↓
Agent: "好的，已记录：
        - 爸爸：不吃辣
        - 孩子：不吃苦瓜、不吃葱
        - 用餐时间：早7点/午12点/晚6点
        
        请问每个人的身高体重方便告诉我吗？
        这样我可以给出更健康的饮食建议。"
    ↓
用户: "爸爸175/70，妈妈160/55，孩子130/30"
    ↓
Agent: "已记录！家庭画像建立完成。
        现在你可以拍照冰箱，我来帮你管理食材。"
    ↓
(数据写入SQLite family_profile表)
```

### 5.2 流程二：拍照入库

```
用户拍照冰箱
    ↓
前端发送图片到 /api/recognize
    ↓
后端调用MiMo-v2.5（多模态模型）
    ↓
Prompt: "请识别图片中的食材，返回JSON格式：
        name, category, quantity, unit, 
        production_date, expiry_date, confidence"
    ↓
MiMo返回识别结果
    ↓
写入SQLite fridge_inventory表
    ↓
返回前端："识别到3种食材：
        - 鸡蛋 × 6，保质期8月15日
        - 西红柿 × 3，保质期8月5日
        - 牛奶 × 1，保质期8月10日
        已存入冰箱库存。"
```

### 5.3 流程三：定时推荐

```
APScheduler定时检查
    ↓
发现距离最近用餐时间还有1-2小时
    ↓
查询family_profile（获取忌口）
查询fridge_inventory（获取库存）
    ↓
构建Prompt发给MiMo-pro：
"基于以下信息推荐晚餐菜谱：
 
【家庭画像】
- 爸爸：不吃辣
- 孩子：不吃苦瓜、不吃葱
 
【冰箱库存】
- 鸡蛋 × 6
- 西红柿 × 3
- 西兰花 × 2
 
请生成A/B两套方案，每套包含2-3道菜。
每道菜注明：菜名、所需食材、烹饪时间、
需要跳过的配料及原因。"
    ↓
MiMo返回A/B方案
    ↓
写入SQLite meal_plan表
    ↓
推送前端通知："晚餐推荐已生成，请查看！"
```

### 5.4 流程四：确认扣减

```
用户在前端查看A/B方案
    ↓
用户选择："选A方案"
    ↓
前端发送到 /api/confirm
    ↓
后端解析A方案所需食材
    ↓
从fridge_inventory扣减对应数量
    ↓
更新meal_plan状态为confirmed
    ↓
返回："已选择A方案，库存已更新：
       - 西红柿：3→1
       - 鸡蛋：6→4
       - 西兰花：2→0"
```

### 5.5 流程五：动态更新画像

```
用户: "我最近在减肥，少油少盐"
    ↓
Agent: (调用MiMo-pro理解意图)
    ↓
Agent: "好的，已更新你的饮食偏好：
        - 新增：少油、少盐
        
        后续推荐菜谱会优先选择清淡的做法。
        还有其他需要调整的吗？"
    ↓

【场景1：首次建档】（1分钟）
├── 打开App，进入对话界面
├── 输入："我家3口人，爸爸不吃辣，孩子不吃苦瓜和葱"
├── Agent回复确认信息
├── 补充身高体重
└── Agent："家庭画像建立完成！"

【场景2：拍照入库】（2分钟）
├── 点击拍照按钮
├── 拍摄冰箱照片（提前准备好食材）
├── Agent识别出食材列表
├── 确认入库
└── 查看库存："冰箱现在有什么？"

【场景3：菜谱推荐】（2分钟）
├── 说"该做晚饭了"
├── Agent确认用餐人数
├── 生成A/B两套方案
├── 展示方案详情（菜名、食材、时间）
└── 选择A方案

【场景4：库存扣减】（1分钟）
├── 确认选择
├── Agent显示扣减结果
├── 查看更新后的库存
└── 展示食材减少

【场景5：动态更新】（1分钟）
├── 说"我最近在减肥，少油少盐"
├── Agent更新画像
├── 再次推荐菜谱
└── 展示推荐结果更清淡
```
